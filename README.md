# Factory of Hypotheses MVP

MVP "Фабрики гипотез" для диагностики потерь металлов, генерации гипотез улучшения и их оценки на детерминированном движке.

Проект собирает в одном контуре:

- диагностику потерь по матрице "класс крупности x минеральная форма x металл";
- генерацию гипотез по проблемным ячейкам;
- ранжирование гипотез по эффекту, реализуемости и покрытию;
- контур обучения на результатах экспериментов;
- RAG по локальному корпусу литературы через Ollama.

LLM в проекте не считает экономический эффект и не подменяет движок. Его роль - помогать с разбором входных данных, формировать текст карточки гипотезы и работать с литературным корпусом.

## Что внутри

- `backend/` - FastAPI API, движок оценки, ingest, RAG-модуль.
- `frontend/` - Streamlit-интерфейс для демо и ручной проверки сценариев.
- `db/` - схема и demo seed для PostgreSQL.
- `scripts/` - вспомогательные скрипты.
- `start.bat` / `stop.bat` - запуск и остановка проекта в Windows.
- `docker-compose.yml` - локальный стек `db + api + frontend + tools`.

## Требования

Минимум для основного сценария:

- Docker Desktop с запущенным engine;
- Windows PowerShell или `cmd`;
- свободные порты `8000`, `8501`, `5433`.

Для вкладки с литературой и локального RAG дополнительно нужен Ollama:

- установленный и запущенный Ollama;
- модель эмбеддингов `granite-embedding:278m`;
- чат-модель `qwen2.5:3b`.

Команды для подготовки Ollama:

```powershell
ollama pull granite-embedding:278m
ollama pull qwen2.5:3b
```

Если Ollama не запущен, основной движок всё равно работает, но RAG-вкладка и литературные гипотезы будут недоступны.

## Быстрый запуск

Самый простой путь - запустить проект через `start.bat`.

Что делает скрипт:

1. Проверяет Docker.
2. Создаёт `.env` из `.env.example`, если файла ещё нет.
3. Проверяет доступность Ollama и показывает предупреждение, если его нет.
4. Поднимает контейнеры `db`, `api`, `frontend`.
5. Заливает demo-данные.
6. Индексирует литературный корпус.
7. Ждёт готовности API и открывает UI в браузере.

Запуск:

```powershell
.\start.bat
```

Остановка:

```powershell
.\stop.bat
```

После старта будут доступны:

- UI: `http://localhost:8501`
- API: `http://localhost:8000`
- Swagger: `http://localhost:8000/docs`

## Запуск через Docker вручную

Если нужен контроль по шагам:

```powershell
docker compose up -d
docker compose --profile tools run --rm seed
docker compose --profile tools run --rm corpus-index
```

Полезные команды:

```powershell
docker compose logs -f api
docker compose logs -f frontend
docker compose down
```

Примечания:

- база данных поднимается на `localhost:5433`;
- корпус монтируется из папки `Задача 1`;
- контейнеры используют `host.docker.internal:11434` для доступа к Ollama на хосте.

## Локальный запуск без Docker

Подходит, если хочешь запускать API напрямую, но PostgreSQL всё равно должен быть доступен по `DATABASE_URL`.

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
python -m db.seed
uvicorn backend.main:app --reload
```

Frontend можно запустить отдельно:

```powershell
streamlit run frontend/app.py
```

По умолчанию UI ожидает API на `http://localhost:8000`.

## Переменные окружения

Основные настройки лежат в `.env.example`.

Ключевые переменные:

- `DATABASE_URL` - строка подключения к PostgreSQL;
- `OPENROUTER_API_KEY` - ключ OpenRouter, если используется внешний LLM-контур;
- `OPENROUTER_MODEL` - модель OpenRouter;
- `OLLAMA_BASE_URL` - адрес локального Ollama;
- `OLLAMA_EMBED_MODEL` - модель эмбеддингов для RAG;
- `OLLAMA_CHAT_MODEL` - чат-модель для ответов по корпусу;
- `EMBED_DIM` - размерность эмбеддингов;
- `CORPUS_PATH` - путь к литературному корпусу;
- `RAG_TOP_K` - сколько фрагментов брать в retriever.

Для большинства локальных запусков хватает значений по умолчанию из `.env.example`.

## Типовой демо-сценарий

1. Открой `http://localhost:8501`.
2. На вкладке "Диагноз" проверь, где сосредоточены основные потери.
3. На вкладке "Гипотезы" сгенерируй идеи для тяжёлых ячеек или добавь экспертную гипотезу.
4. На вкладке "Рейтинг" оцени фабрику и посмотри ранжирование.
5. На вкладке "Карточка" собери текстовое обоснование гипотезы.
6. На вкладке "Лаборатория" загрузи результат эксперимента и проверь, как меняется коэффициент правила.
7. На вкладке "Литература" задай вопрос к корпусу или предложи гипотезы из литературы.

## Основные API-эндпоинты

- `GET /health` - статус API.
- `GET /plants` - список фабрик.
- `POST /plants/ingest` - загрузка XLSX с матрицей потерь.
- `GET /plants/{plant_id}/diagnosis` - диагноз по потерям.
- `POST /plants/{plant_id}/generate` - генерация гипотез.
- `POST /plants/{plant_id}/evaluate` - оценка гипотез по фабрике.
- `GET /plants/{plant_id}/ranking` - рейтинг гипотез.
- `POST /hypotheses/{hypothesis_id}/roadmap` - построение дорожной карты.
- `POST /roadmap/{step_id}/artifact` - загрузка результата эксперимента.
- `POST /corpus/index` - индексация корпуса.
- `GET /corpus/search` - поиск по корпусу.
- `POST /corpus/ask` - ответ по корпусу с цитированием источников.

## Структура данных и логика

В проекте уже реализованы:

- схема БД под MVP и demo seed;
- правила движка для модулей `regrind`, `classification`, `fine_flotation`;
- оценка эффекта, покрытий и конфликтов гипотез;
- контур dead-end и калибровки коэффициентов;
- индексатор PDF/DOCX-корпуса в pgvector через Ollama.

## Если что-то не стартует

- Проверь, что Docker Desktop действительно запущен.
- Проверь, что порты `5433`, `8000`, `8501` не заняты.
- Если не работает литература, проверь `http://localhost:11434/api/tags`.
- Если API не поднялся, смотри `docker compose logs -f api`.
- Если UI пустой, повторно выполни `docker compose --profile tools run --rm seed`.

## Текущее состояние

Проект ориентирован на локальный MVP-демо контур: сначала seed и ручная проверка сценариев, потом доработка правил, данных и корпуса под реальные фабрики.
