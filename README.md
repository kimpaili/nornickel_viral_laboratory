# Factory of Hypotheses MVP

MVP "Фабрика гипотез" для диагностики потерь металлов, генерации гипотез улучшения, их оценки на детерминированном движке и работы с литературным корпусом через локальный RAG.

Проект нужен для демонстрации следующего контура:

- есть фактические потери металла в хвостах;
- система показывает, где именно сосредоточены потери;
- по проблемным зонам генерируются гипотезы улучшения;
- гипотезы оцениваются на правилах и числах, а не "на вкус LLM";
- результаты опытов можно загрузить обратно и скорректировать поведение системы;
- литературный корпус помогает искать обоснования и дополнительные идеи через RAG.

Важно: LLM в этом проекте не подменяет расчётный движок. Все оценки эффекта, ранжирование, конфликты и логика "что выгоднее / реализуемее" считаются детерминированным backend. LLM здесь помогает формулировать текст, работать с литературой и в отдельных местах улучшать UX.

## Что умеет проект

- Диагностика потерь по матрице `класс крупности x минеральная форма x металл`.
- Генерация гипотез по проблемным ячейкам.
- Оценка гипотез по правилам, коэффициентам и доступности оборудования.
- Ранжирование гипотез по эффекту, покрытию и конфликтам.
- Формирование карточки гипотезы в текстовом виде.
- Построение дорожной карты для эксперимента.
- Контур "лаборатория -> результат опыта -> обучение / подтверждение / тупик".
- Поиск по локальному корпусу PDF/DOCX и ответы по нему через RAG.

## Архитектура

Проект состоит из четырёх основных частей:

- `backend/` - FastAPI API, бизнес-логика, движок оценки, генерация гипотез, RAG, работа с БД.
- `frontend/` - Streamlit UI для демонстрации сценариев.
- `db/` - схема базы и demo seed.
- `docker-compose.yml` - локальный стек сервисов.

Локальный Docker-стек поднимает:

- `db` - PostgreSQL с `pgvector`;
- `api` - FastAPI backend;
- `frontend` - Streamlit UI;
- `seed` - одноразовый сервис для наполнения demo-данными;
- `corpus-index` - одноразовый сервис для индексации корпуса литературы.

## Как устроена логика продукта

Типовой поток работы такой:

1. Пользователь выбирает фабрику.
2. Система показывает диагностику потерь: где теряется металл и какие потери извлекаемы.
3. По этим данным можно сгенерировать гипотезы или добавить экспертную гипотезу вручную.
4. Для гипотез выполняется оценка на детерминированных правилах.
5. Система ранжирует гипотезы и показывает конкуренцию за одни и те же потери.
6. Для выбранной гипотезы можно собрать текстовую карточку.
7. Для гипотезы можно построить roadmap эксперимента и загрузить результат опыта.
8. На вкладке литературы можно искать подтверждения и идеи в локальном корпусе.

## RAG: как он работает в этом проекте

RAG в этом проекте полностью опирается на локальный корпус документов.

Схема работы такая:

1. Из папки корпуса читаются PDF и DOCX.
2. Документы режутся на фрагменты.
3. Для фрагментов строятся эмбеддинги через локальный Ollama.
4. Эмбеддинги сохраняются в PostgreSQL с `pgvector`.
5. При запросе пользователя система ищет ближайшие фрагменты.
6. Дальше либо возвращаются найденные куски, либо формируется ответ через чат-модель Ollama.

RAG используется только для литературных обоснований и поиска по корпусу. Он не участвует в расчёте числового эффекта гипотез.

## Какие файлы лучше класть в корпус

Практически полезно складывать туда:

- PDF-книги и статьи;
- DOCX-заметки и методички;
- документы, из которых реально стоит извлекать фрагменты для поиска.

Не стоит класть в корпус:

- случайные большие бинарники;
- архивы;
- служебные файлы, не относящиеся к литературе;
- документы, которые не нужно показывать в поиске и цитировании.

## Быстрый запуск через `start.bat`

Если нужен самый простой способ старта, используй:

```powershell
.\start.bat
```

Скрипт делает следующее:

1. Проверяет, что Docker доступен.
2. Переходит в корень проекта.
3. Создаёт `.env` из `.env.example`, если файла ещё нет.
4. Проверяет доступность `http://localhost:11434/api/tags`.
5. Если Ollama не найден, показывает предупреждение, но не останавливает запуск.
6. Выполняет `docker compose up -d`.
7. Запускает сервис `seed`.
8. Запускает сервис `corpus-index`.
9. Ожидает готовность API по `GET /health`.
10. Открывает UI в браузере на `http://localhost:8501`.

После успешного запуска доступны:

- UI: `http://localhost:8501`
- API: `http://localhost:8000`
- Swagger: `http://localhost:8000/docs`

## Что делает `stop.bat`

Остановка проекта:

```powershell
.\stop.bat
```

Скрипт выполняет:

- проверку Docker;
- `docker compose down`.

При этом:

- контейнеры останавливаются;
- сеть и контейнеры удаляются;
- данные PostgreSQL сохраняются в Docker volume;
- папка корпуса `Задача 1` остаётся на месте.

Полный сброс данных:

```powershell
docker compose down -v
```

Эта команда удалит и volume базы.

## Как Docker-настройки устроены внутри

Фактические параметры в `docker-compose.yml` такие:

- База поднимается как `pgvector/pgvector:pg16`.
- PostgreSQL пробрасывается на `localhost:5433`.
- API работает на `0.0.0.0:8000`.
- Frontend работает на `0.0.0.0:8501`.
- Для контейнеров `api` и `corpus-index` Ollama доступен как `http://host.docker.internal:11434`.
- Папка `./Задача 1` монтируется в контейнеры как `/corpus`.

Это важный момент:

- на хосте в `.env` используется `OLLAMA_BASE_URL=http://localhost:11434`;
- внутри Docker это значение не используется напрямую для backend-контейнера;
- в compose оно специально переопределено на `http://host.docker.internal:11434`, чтобы контейнер видел Ollama на хосте.

## Локальный запуск без Docker

Такой режим тоже возможен, но он сложнее и нужен в основном для разработки.

Важно понимать ограничение:

- backend ожидает PostgreSQL с `pgvector`;
- SQLite или "просто локальный файл" здесь не подойдут;
- значит даже без Docker нужен доступный PostgreSQL по `DATABASE_URL`.

### Вариант 1. Локально Python, база в Docker

Это самый практичный dev-режим.

Сначала подними только БД через Docker:

```powershell

```

Потом создай окружение и установи зависимости:

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
```

После этого можно выполнить seed:

```powershell
python -m db.seed
```

Запуск backend:

```powershell
uvicorn backend.main:app --reload
```

Отдельно запуск frontend:

```powershell
streamlit run frontend/app.py
```

### Вариант 2. Полностью локальный запуск без Docker

Он возможен только если у тебя уже есть:

- локальный PostgreSQL;
- установленный `pgvector`;
- созданная БД `factory_hypotheses`;
- корректная строка подключения в `DATABASE_URL`.

Плюс, если нужен RAG:

- локальный Ollama;
- папка корпуса;
- скачанные модели.

## Переменные окружения

Основные настройки лежат в `.env.example`.

Текущие ключевые переменные:

- `DATABASE_URL` - строка подключения к PostgreSQL.
- `OPENROUTER_API_KEY` - ключ OpenRouter.
- `OPENROUTER_BASE_URL` - URL OpenRouter API.
- `OPENROUTER_MODEL` - модель OpenRouter.
- `OPENROUTER_APP_TITLE` - имя приложения для заголовков OpenRouter.
- `OPENROUTER_APP_URL` - URL приложения для заголовков OpenRouter.
- `OLLAMA_BASE_URL` - адрес локального Ollama для запуска на хосте.
- `OLLAMA_EMBED_MODEL` - модель эмбеддингов для RAG.
- `OLLAMA_CHAT_MODEL` - чат-модель для RAG и текстов.
- `OLLAMA_TIMEOUT` - timeout запросов к Ollama.
- `EMBED_DIM` - размерность эмбеддингов.
- `CORPUS_PATH` - путь к папке корпуса.
- `RAG_TOP_K` - сколько фрагментов брать из retriever.

Замечания:

- для большинства сценариев достаточно значений из `.env.example`;
- если запускаешь backend локально, он читает `.env` через `python-dotenv`;
- если запускаешь через Docker Compose, часть значений берётся из `.env`, а часть переопределяется в `docker-compose.yml`.

## OpenRouter: где он нужен, а где нет

В проекте OpenRouter не нужен для основного числового контура.

Основная идея такая:

- расчёт эффекта гипотез не зависит от OpenRouter;
- локальный RAG работает через Ollama;
- OpenRouter оставлен как внешний LLM-контур для отдельных вспомогательных сценариев.

Поэтому:

- если `OPENROUTER_API_KEY` пустой, проект всё равно может работать;
- для demo-сценария важнее Docker, PostgreSQL, corpus и Ollama.

## Основные API-эндпоинты

Базовые:

- `GET /health` - статус API и проверка готовности.
- `GET /plants` - список фабрик.

Гипотезы и диагностика:

- `GET /plants/{plant_id}/diagnosis` - диагностика потерь.
- `POST /plants/{plant_id}/generate` - генерация гипотез.
- `GET /plants/{plant_id}/hypotheses` - список гипотез по фабрике.
- `POST /hypotheses/ingest` - добавление экспертной гипотезы.
- `POST /plants/{plant_id}/evaluate` - оценка гипотез.
- `GET /plants/{plant_id}/ranking` - ранжирование гипотез.
- `GET /hypotheses/{hypothesis_id}/card` - карточка гипотезы.

Дорожная карта и обучение:

- `POST /hypotheses/{hypothesis_id}/roadmap` - построение дорожной карты.
- `POST /roadmap/{step_id}/artifact` - загрузка результата эксперимента.

Литература и RAG:

- `POST /plants/{plant_id}/literature-hypotheses` - идеи из литературы.
- `GET /corpus/stats` - статистика корпуса и состояние Ollama.
- `POST /corpus/index` - индексация корпуса.
- `GET /corpus/search` - поиск по корпусу.
- `POST /corpus/ask` - ответ по корпусу с опорой на найденные фрагменты.

Полный список и схемы запросов доступны в Swagger:

- `http://localhost:8000/docs`

## Что происходит при индексации корпуса

Когда выполняется `corpus-index` или `POST /corpus/index`, backend:

- обходит папку корпуса;
- читает документы;
- разбивает документы на фрагменты;
- считает хэш файлов и старается работать идемпотентно;
- получает эмбеддинги через локальный Ollama;
- сохраняет документы и фрагменты в таблицы корпуса;
- использует `pgvector` для последующего semantic search.

Поэтому при проблемах с индексированием нужно смотреть не только на API, но и на:

- доступность папки корпуса;
- доступность Ollama;
- наличие нужной embedding-модели;
- состояние PostgreSQL.

Отдельно проверь:

- доступна ли БД;
- корректен ли `DATABASE_URL`;
- не сломан ли импорт зависимостей.

### Нет данных в интерфейсе

Скорее всего не выполнен seed.

Запусти:

```powershell
docker compose --profile tools run --rm seed
```

### Не работает RAG / вкладка литературы

Проверь по порядку:

1. Есть ли папка `Задача 1` в корне проекта.
2. Есть ли в ней документы.
3. Доступен ли Ollama по `http://localhost:11434/api/tags`.
4. Скачаны ли модели `granite-embedding:278m` и `qwen2.5:3b`.
5. Выполнялась ли индексация корпуса.

### Индексация корпуса падает

Проверь:

- существует ли `./Задача 1`;
- не пустая ли папка;
- не содержит ли она неожиданные битые файлы;
- доступен ли Ollama;
- есть ли модель эмбеддингов;
- жив ли PostgreSQL.

### Порты заняты

Проект использует:

- `5433` - PostgreSQL
- `8000` - API
- `8501` - Streamlit UI
- `11434` - Ollama на хосте

Если один из портов занят, либо освободи его, либо меняй конфигурацию осознанно.

## Для разработки

Если планируются изменения в коде, практичный цикл такой:

1. Поднять `db` через Docker.
2. Запускать backend локально с `--reload`.
3. Запускать frontend локально отдельной командой.
4. По мере необходимости пересевать БД и переиндексировать корпус.

Почему так удобно:

- быстрее итерации;
- проще дебажить Python-код;
- сохраняется совместимость с реальной БД и `pgvector`.

## Ограничения текущего MVP

Важно держать в голове, что это именно MVP:

- правила и коэффициенты зашиты в demo-логике;
- seed-данные синтетические;
- набор модулей ограничен;
- RAG работает по локальной папке, а не по полноценному knowledge platform;
- часть LLM-сценариев сделана как вспомогательный слой поверх основного детерминированного контура.

## Коротко: рекомендуемый способ запуска

Если нужен самый надёжный путь, делай так:

1. Установи Docker Desktop.
2. Установи Ollama.
3. Скачай модели:

```powershell
ollama pull granite-embedding:278m
ollama pull qwen2.5:3b
```

4. Проверь, что в корне проекта есть папка `Задача 1`.
5. Запусти:

```powershell
.\start.bat
```

6. Открой:

- `http://localhost:8501`

Этого достаточно для полного локального demo-сценария, включая RAG.
