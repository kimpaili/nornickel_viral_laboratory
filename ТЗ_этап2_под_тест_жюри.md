# ТЗ — этап 2. Доводка MVP под тестирование жюри

*Ключевое изменение: жюри будет ТЕСТИРОВАТЬ развёрнутый продукт руками. Приоритет смещается с «сколько фич» на «что увидит и потрогает тестировщик». Лучше меньше функций, но каждая работает с первого клика и наглядна. Дедлайн: 4 июля 23:59.*

## Принцип этапа
Каждая задача оценивается вопросом: **«что жюри увидит, когда нажмёт на это?»** Фича без наглядного, самообъясняющего выхода не имеет ценности на тесте. Ничего не должно падать на их данных и их порядке действий.

## Приоритет P0 — без этого нельзя выходить (надёжность теста)

1. **Приём DOCX и PDF, не только XLSX.** Их гипотезы — в DOCX, схемы и оборудование — картинками/PDF. Если жюри загрузит DOCX и не сработает — провал на ровном месте. DOCX-гипотезы → `hypothesis(origin=expert)`. PDF/текст → в корпус. (Тихонов.)
2. **Наполнить базу знаний реальным содержимым.** Без этого RAG пустой и обоснования голые. Три источника: (1) файлы ТЗ и данные кейса, (2) рекомендованные в ТЗ (учебник по обогащению, статьи), (3) 3–5 своих проверенных источников по геометаллургии/флотации. Проиндексировать в корпус до демо. (Полина готовит подборку → Тихонов индексирует.)
3. **Предзагруженный демо-сценарий.** На развёрнутом стенде должны быть УЖЕ загружены 2 фабрики с готовыми матрицами и гипотезами, чтобы жюри сразу тыкало, а не настраивало. Кнопка «загрузить пример». Плюс пустой режим для их собственной загрузки. (Быков.)
4. **Обработка кривых входов без падений.** Пустые ячейки, лишние листы, кривой формат — не 500-я ошибка, а понятное сообщение. Жюри специально грузит мусор. (Тихонов.)

## Приоритет P1 — то, что даёт «вау» на тесте (наглядность)

5. **Мини-отчёт по каждому модулю (прозрачность).** Главная фича этапа. По каждой гипотезе тестировщик может раскрыть каждый модуль и увидеть его логику: на какие ячейки матрицы бил, какое правило применил, какой коэффициент, какие числа фабрики взял, что на выходе. Приоритет гипотезы виден как сумма вкладов модулей. Это операционализирует «не чёрный ящик» из ТЗ вживую. (Быков — UI, Тихонов — отдать данные модулей в API.)
6. **Матрица потерь как интерактивная тепловая таблица** на экране диагноза: класс × форма, подсветка тяжёлых ячеек, клик по ячейке → какие гипотезы её закрывают. Это визуальный якорь всего продукта. (Быков.)
7. **Межфабричный перенос тупиков — на витрину.** Когда гипотеза совпала с известным тупиком, показать плашку «уже проверялось на фабрике X, результат отрицательный, причина …». Данные уже есть (тупик привязан к модуль+причина+класс), нужен UI. (Быков.)
8. **Конфликт гипотез за ячейку — явно.** В карточке conflict-flag; в портфеле строка «X и Y конкурируют за Z т, эффект не суммируется». Логика обрезки уже есть — вывести наружу. (Тихонов — логика, Быков — показ.)
9. **Дорожная карта визуально схемой.** Рендерим граф этапов детерминированно из данных (react-flow/mermaid), с задачами по этапам и подсветкой сращённых узлов. НЕ доверять отрисовку LLM. (Быков.)

## Приоритет P2 — закрыть пункты ТЗ, если есть время

10. **Экспорт CSV из БД (быстро, поднять в P1).** Данные уже в БД — сериализовать результат запроса в CSV, отдать на скачивание. Точки:
    - `GET /export/portfolio.csv?plant_id=` → портфель фабрики: JOIN `hypothesis` + `evaluation` (+ `module`), колонки: hypothesis_id, title, origin, module, target_cause, effect_tons_min/max, effect_usd_min/max, feasible, relevance_score, rank, dead_end_flag. Источник — таблица `evaluation`.
    - `GET /export/hypothesis/{id}.csv` → одна гипотеза с разбивкой по модулям и этапами дорожной карты (`roadmap_step`).
    - `GET /export/matrix.csv?plant_id=` → матрица потерь из `loss_cell` (класс × форма × металл × тонны).
    Реализация: FastAPI `StreamingResponse` + `csv.writer`, `Content-Disposition: attachment`. Кнопка «Экспорт CSV» на экранах рейтинга, карточки и диагноза.
11. **Экспорт PDF дашбордов с сайта.** Тестировщик жмёт «Скачать PDF» на текущем дашборде (портфель / карточка / диагноз) → получает файл «как на экране». Реализация клиентская, чтобы PDF совпадал с UI: `html2canvas` снимает целевой контейнер дашборда → `jsPDF` кладёт в PDF (multi-page для длинных). Каждый экспортируемый экран оборачиваем в контейнер с id (`#dash-portfolio`, `#dash-card`, `#dash-diagnosis`), кнопка экспорта рендерит именно его. Подключить печатный CSS (скрыть навигацию/кнопки в снимке). Библиотеки с cdnjs. (Быков — клиент; данные уже на экране.)
12. **Редактор правил:** эксперт меняет коэффициент → пересчёт рейтинга на глазах. Закрывает «экспертную настройку» + «обучение на фидбэке». Мощно на тесте, но только если не ломает пересчёт. (Тихонов.)
13. **Оптимизация портфеля экспериментов:** при нескольких гипотезах на проверку — объединённый план по общим этапам, ранжированный по выгода/стоимость. Держать узко: жадное объединение + сортировка, не полный оптимизатор. (Быков + мат-часть, см. §14.)

## §14. Мат-модель (Быков) — научно обоснованная

**Основное: кривая извлечения по крупности и раскрытию** как физоснова модулей флотации/доизмельчения. Классика (Gaudin 1939; Trahar 1981; «elephant curve»): извлечение падает на тонких и крупных частицах, максимум ~10–100 мкм, зависит от степени раскрытия. Функциональная форма (напр. Gamma/логнормаль для зависимости от крупности + линейная по раскрытию) с параметрами, **калибруемыми на данных фабрики**. Это превращает коэффициент модуля из плоского числа в обоснованную кривую и защищено учебником — эксперты Норникеля это знают с вуза.
- Обязательно подать как «известная форма + калибровка на ваших данных», НЕ как first-principles предсказание.
- Не лезть в population balance для измельчения (закопают без данных калибровки).

**Дополнительно (если время): калибровка коэффициента как взвешенное байесовское обновление** — шаг тем меньше, чем больше накоплено экспериментов. Привязывается к петле обучения, звучит убедительно для комитета.

## §15. Доработки концепта (Полина вносит в документ)
- Политика конфликта за ячейку (conflict-flag / нет суммирования).
- Формула калибровки (взвешенное обновление, шаг ∝ 1/√N экспериментов).
- Межфабричный перенос тупиков и правил — вынести как отдельную сильную фичу масштабируемости.
- Конфиденциальность корпуса: on-prem/закрытый контур LLM для патентов и внутренних отчётов.
- Приоритет правил: специфичное (по классу крупности) важнее общего — как часть контракта модуля.

## §16. Фронтенд — спецификация (переделать нагенеренный UI)

**Принцип: сложность — в структуре, простота — в каждом экране. Минимум текста, глубина по клику.**

**Каркас.** Слева — вертикальный «позвоночник» из 6 стадий (всегда виден = масштаб системы): Диагноз → Гипотезы → Оценка → Карточка → Дорожная карта → Лаборатория. Справа — активная стадия. Сверху — тонкая панель: переключатель фабрики, «Загрузить пример», «Загрузить свои данные», экспорт CSV/PDF. Больше в шапке ничего.

**Три правила против каши:**
1. Одно действие на экран. Диагноз — только матрица. Рейтинг — только список. Никаких экранов «всё сразу».
2. Глубина через раскрытие, не через текст. Строка гипотезы свёрнута до сути; клик раскрывает мини-отчёты модулей с provenance. Вся сложность живёт внутри раскрытий — тестировщик сам выбирает глубину.
3. Один визуальный язык. Три цвета матрицы (бирюза — свободный/флотация, янтарь — заперт/измельчение, серый — потерян) используются везде: матрица, чипы гипотез, модули, карта покрытия. Понял язык один раз — читаешь всё без подписей.

**Экраны:**
- **Диагноз** (`#dash-diagnosis`): матрица потерь тепловой таблицей (класс × форма), метрики сверху (потеряно т, извлекаемо %). Клик по ячейке → подсветка гипотез, которые её закрывают.
- **Гипотезы**: список карточек-строк (название, чип целевой ячейки в цвете формы, бейдж origin expert/generated, быстрый эффект). Тумблер режима «оцени мои / предложи».
- **Оценка / Рейтинг** (`#dash-portfolio`): отранжированный список. Каждая строка раскрывается в **мини-отчёты модулей** (на какие ячейки бил, правило, коэффициент, числа фабрики, вклад в приоритет). Плашки: тупик «уже проваливалось на фабрике X», конфликт «конкурирует с Y за Z т». Карта покрытия. Переключатель фабрики здесь = killer-демо A.
- **Карточка** (`#dash-card`): одна гипотеза целиком — эффект (т и $ диапазоном), вердикты модулей с provenance, killer-эксперимент. Экспорт PDF/CSV.
- **Дорожная карта**: граф этапов (react-flow/mermaid, детерминированно из данных), сращённые узлы подсвечены, задачи по этапам.
- **Лаборатория**: загрузка результата эксперимента → на глазах меняется коэффициент, появляется тупик → возврат на «Гипотезы» показывает, что тупик отфильтрован. Killer-демо B.

**Технически:** экспортируемые экраны обёрнуты в контейнеры с id (см. §11). Никаких длинных абзацев в UI — только числа, чипы, короткие подписи. Цвета — через CSS-переменные, светлая/тёмная тема.

## §17. Definition of Done для теста жюри
- Развёрнутый стенд, 2 фабрики предзагружены, «загрузить пример» работает.
- DOCX и PDF принимаются без падений; мусор даёт понятное сообщение, не 500.
- По каждой гипотезе раскрывается мини-отчёт каждого модуля с provenance.
- Матрица кликабельна; тупик и конфликт показываются явно.
- Ни одного числа оценки от LLM — всё из движка (главный аргумент прозрачности).
- Экспорт CSV из БД и PDF дашборда работают с кнопки.
- Фронт по §16: позвоночник из 6 стадий, один визуальный язык, глубина по клику.
- README с одной командой запуска; видео-демо ≤5 мин по сценарию.

## §18. Кто что ведёт
- **Тихонов:** P0-приём DOCX/PDF, наполнение корпуса, устойчивость к мусору, данные модулей в API, конфликт-логика, экспорт CSV из БД, редактор правил.
- **Быков:** фронт по §16, предзагруженный сценарий, мини-отчёты модулей, интерактивная матрица, витрина тупиков, визуальная дорожная карта, экспорт PDF дашбордов, мат-модель (§14), оптимизация портфеля.
- **Полина:** подборка источников для корпуса, доработки концепта (§15), научное обоснование мат-части, презентация, речь, репетиция демо-сценария под тест жюри.
